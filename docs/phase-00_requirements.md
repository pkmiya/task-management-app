## 4. Phase 01 — 機能要件（ユースケース）

| ID | ユースケース |
|----|----------------|
| UC-01 | タスクを作成する |
| UC-02 | タスク一覧を取得する |
| UC-03 | タスクを 1 件取得する（ID 指定） |
| UC-04 | タスクのタイトルを変更する |
| UC-05 | タスクを削除する |

**備考**: 「タスクを完了する」および **未着手／完了などのステータス概念・状態遷移**は Phase 01 の対象外とし、後フェーズで要件化する（セクション 8 参照）。

### 4.1 一覧取得（UC-02）

- **全件**を返す。
- **並び順**: **作成日時の昇順（作成順）を第一キー**とし、同一タイムスタンプのときは **タスク ID の昇順を第二キー**とする（決定的な並びを保証する）。他の並び・フィルタは Phase 04 以降で検討。
- **ページネーションなし**。
- タスクが **0 件**のときは **成功** とし、**空のコレクション**を返す（HTTP に載せる場合は Phase 02 で **200 相当 + 空配列** でよい）。

### 4.2 一覧・詳細の返却形（UC-02・UC-03）

- Phase 01 はシンプルさ優先で、一覧・詳細とも **同一のタスク表現（DTO）** でよい。
- 一覧のみ軽量フィールドにするなどの改善は **後フェーズ**で検討する。
- **ユースケース層の戻り値の型**（ドメインの `Task` そのものか、アプリケーション内部用の構造体か）は実装で選び、理由を説明できればよい。**HTTP の JSON 用リクエスト／レスポンス DTO は Phase 02** でハンドラ側に置く（Phase 01 の要件からは除外する）。

### 4.3 利用者前提

- **単一ユーザー**のみを想定する（複数ユーザー・テナントは扱わない）。
- **ユーザーの認証・認可およびユーザー CRUD は実装しない**（Phase 04 以降も**実装予定は未定**）。

### 4.4 タイトル変更（UC-04）

- **フロー**: 指定 ID で **既存タスクを取得**し（存在しなければ **NotFound**）、ドメイン上で **タイトルを変更**し、変更後のエンティティを **`Save` で永続化**する。
- **作成（UC-01）** と **更新（UC-04）** はユースケースとして分けるが、永続化はいずれも **`Save`** を通じてよい（セクション 5.6 と整合）。

## 5. Phase 01 — ドメインルール・データ項目

### 5.1 識別子

- タスク ID は **UUID**。
- **生成は「ドメインでは値として表現し、生成責務はAggregate外のどこが適切か」設計判断の対象とする**（典型的にはリポジトリ生成時やファクトリ／ユースケース等。実装時に選定し、理由を説明できること）。

### 5.2 タイトル（UC-04 含む作成時）

- **トリム**: 入力タイトルは **先頭・末尾の Unicode 空白をトリムしたうえで** 検証する（どのコードポイントをトリム対象とするかは実装で一つに決め、一貫して適用すること）。
- **トリム後**の文字列が **空**であることは不可。
- **トリム後**が **Unicode の空白文字のみ**からなることも不可（例: スペース・タブのみなど）。空白の判定は実装で一つに決め（例: Go の `unicode.IsSpace` で各コードポイントを検査し、すべて空白のみなら不正とする）、一貫して適用すること。
- **最大長 255** は **トリム後**の文字列について **Unicode コードポイント数**で数える。**改行を含むタイトルも許可**する（255 コードポイント以内であればよい）。
- **他タスクと同一タイトルは許可**する。

### 5.3 ステータス・状態遷移（Phase 01）

Phase 01 のドメインモデルおよびユースケースでは **ステータス（未着手／進行中／完了など）を持たない**。すべてのタスクは **同一種類のエンティティ**として扱う。

後フェーズで状態を導入する場合は **未着手と完了の 2 値のみ** を基本とし、**進行中**や複雑な遷移グラフはそのときに要件化する。**キャンセル**状態・関連ユースケースも Phase 04 以降の候補とする（セクション 8 参照）。

### 5.4 メタデータ（作成日時）

- Phase 01 で保持するメタデータは **作成日時のみ**。**更新日時・削除日時・ソフトデリートフラグなど**は Phase 04 以降の適切なタイミングで検討する。
- **タイムゾーン**: **UTC** で保持する（並び順・比較の一貫性のため）。
- **精度**: Go の **`time.Time`** に準ずればよい（実質ナノ秒精度）。
- **確定タイミング**: UC-01（タスク作成）の処理のうち、**保存される時点で一意に決まっていること**。ユースケース内で付与する／Aggregate の生成処理で渡すなど、責務の置き場は実装で選び、理由を説明できること。
- **現在時刻の取得**: **`time.Now()` を全域で直書きする必要はない**。**Clock（現在時刻プロバイダ）を注入**し、単体テストで固定時刻を返せるようにすることが望ましい（Phase 03 のテスト容易性と整合）。

### 5.5 削除の意味（UC-05）

- Phase 01 の削除は **ハードデリート** とする。in-memory repository から **該当タスクを除去**し、削除後は同一 ID で取得できない。**復元・アーカイブは行わない**。
- **ソフトデリート**は Phase 04 以降で検討する（セクション 8 とセクション 5.4 のメタデータ方針と整合）。

### 5.6 リポジトリの契約（Phase 01）

ユースケースから参照する **Task の永続化抽象**として、インタフェースは「使う側」に近い層に置くなど **プロジェクトのレイヤ方針**に従う。メソッドの入出力は **ドメインの Task／タスク ID** に限定し、in-memory の内部表現や将来の DB 行構造などはドメインに漏らさない。

Phase 01 では少なくとも次の操作を満たすこと（メソッド名・シグネチャ・複数メソッドへの分割は実装でよい）。

| 操作 | 概要 |
|------|------|
| **保存（Save）** | タスクを保存する。**新規作成と更新の両方**を扱う（同一 ID は更新として上書きされればよい）。保存対象はドメイン側で不変条件を満たしていることを前提とする。 |
| **ID で取得（FindByID）** | 指定 ID のタスクを **1 件**返す。存在しないときは **NotFound**（セクション 5.7）として区別できること。 |
| **一覧取得（FindAll）** | **全件**を返す。**並び順はセクション 4.1**（作成日時昇順、タイブレークは ID 昇順）。ソートはリポジトリ実装内でもユースケース側でもよいが、呼び出し元から見た結果は要件どおりであること。 |
| **削除（Delete）** | 指定 ID のタスクを **ハードデリート**する（セクション 5.5）。存在しないときは **NotFound** として区別できること。 |

**備考**:

- 「生成時に ID を割り当てる」など保存前の処理が必要な場合は、リポジトリとユースケースのどちらに置くかは設計判断とし、理由を説明できること（セクション 5.1 と整合）。
- **FindByID／Delete** が対象なしのときの **NotFound** と、ユースケースが検知する **Validation** は、ドメインエラー・結果型・ラップされた error など、実装で一貫させればよい（セクション 5.7）。

### 5.7 エラー種別

**エラーの種類**は実装で表現してよいが、少なくとも次を**区別**できること。

| 種類 | 意味・例 |
|------|-----------|
| **Validation** | 入力・不変条件違反。**タイトル不正**（セクション 5.2 のトリム後における空・空白のみ・最大長超過など）はこの種類とする。HTTP に載せる場合は Phase 02 で **4xx（例: 400）** 等へ対応する想定。 |
| **NotFound** | **存在しない ID** による取得・タイトル変更・削除。HTTP に載せる場合は Phase 02 で **404** 等へ対応する想定。 |

| 状況 | 扱い |
|------|------|
| **存在しない ID** での取得・タイトル変更・削除 | **NotFound** |
| **タイトル不正**（UC-01・UC-04） | **Validation** |

**上記以外**（インフラ・内部不整合・分類しない失敗など）は **実装側の一般の `error` 等で表現してよい**。種別の細分化や HTTP とのマッピング（5xx の扱いなど）は **後フェーズで詳細設計**する。

## 6. 非機能要件・制約

| カテゴリ | 要件 |
|----------|------|
| **実装言語** | Go |
| **永続化** | Phase 01〜03 は **in-memory repository**。DB は Phase 04 以降で検討 |
| **並行性・呼び出しモデル** | **Phase 01**: **単一ワーカーから**ユースケース／リポジトリを呼び出す実装・利用を**約束**する（複数ゴールーチンからの並行呼び出しは想定しない）。この前提のもと **リポジトリにミューテックス等は Phase 01 で必須としない**。**Phase 02**: HTTP により並行アクセスがあり得るため、**in-memory リポジトリ実装に排他制御を入れる**（セクション 3 と整合） |
| **運用・観測性・ゲートウェイ** | **構造化ログ、メトリクス、分散トレーシング**および **HTTP のタイムアウト・リクエストボディサイズ上限**などは **本要件の対象外**とする（必要になれば別途要件化する） |
| **アーキテクチャ** | DDD / Clean Architecture **寄り**の責務分離を意識する |
| **設計のバランス** | **過剰設計は避ける**（説明可能な設計を優先） |
| **進め方** | **最小構成から**段階的に（いきなり完成コード一式は出さない） |
| **動作確認（Phase 01）** | **手動またはスクリプト**でユースケースを通せればよい。**自動テストの整備は Phase 03** を主な場とする（Phase 01 で `go test` を書かないことを禁じるわけではないが、完了定義に含めない） |
