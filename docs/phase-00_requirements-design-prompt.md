Phase 00 の入力用プロンプト（要件整理・設計レビューの起点）。

このリポジトリは、PKSHA PSIチームのSWE実務選考インターンに向けた事前学習用です。

前回のインターンでは、GoとDDDの両方が初めてで、設計・実装判断が実務レベルに届かず落選しました。
今回は、Goで小さなタスク管理APIを作りながら、DDDの基本設計を実務に近い形で練習したいです。

目的は、単に動くAPIを作ることではありません。
以下をできるようになることが目的です。

- GoでAPIを設計・実装できる
- Entity / Value Object / Aggregate / Repository / Usecase の責務を説明できる
- 「このロジックをどこに置くべきか」を判断できる
- 設計意図・トレードオフ・改善余地を言語化できる
- インターン本番でレビューに対して自分の判断を説明できる

題材はタスク管理APIです。

**Phase 01 で扱うユースケースの一覧・詳細は要件書セクション 4 を参照。**  
概要のみ列挙すると次のとおり。

- タスクを作成する（UC-01）
- タスク一覧を取得する（UC-02）
- タスクを 1 件取得する（UC-03）
- タスクのタイトルを変更する（UC-04）
- タスクを削除する（UC-05）

「タスクを完了する」およびステータス（未着手／完了など）・状態遷移は Phase 01 の対象外とし、後フェーズで要件化する（要件書セクション 8）。

制約・フェーズ境界の詳細は要件書セクション 3・6 に従う。概要は次のとおり。

- 言語は Go
- Phase 01〜03 は in-memory repository（DB は Phase 04 以降で検討）
- Phase 01 は HTTP 層なし・単一ワーカーからの呼び出しのみなど（詳細は要件書）
- Phase 02 で HTTP を載せる際は、並行アクセスに備え in-memory 実装へ排他制御を入れる（要件書）
- DDD / Clean Architecture 寄りの責務分離を意識する。ただし過剰設計は避ける
- 最小構成から段階的に進める（いきなり完成コードを全部出さない）

Phase 00 ではコード実装は行わない（要件書セクション 3）。

設計レビューで段階的に扱いたい論点は以下。**すべてを一度に書き切る必要はない**（要件書フェーズ計画のとおり）。

1. タスク管理APIの要件を、DDD練習用に整理する。**単一ソースは [`phase-00_requirements.md`](./phase-00_requirements.md)。新規整理だけでなく改版するときもこの文書を正として編集する。**
2. Entity / Value Object / Aggregate / Repository / Usecase の候補を出してください
3. 最小のディレクトリ構成案を出してください
4. 最初に実装すべきユースケースを1つ選んでください（要件書および実装方針では UC-01「タスク作成」を先行することが前提）
5. その設計判断の理由とトレードオフを説明してください

コードを書く前に、まず設計方針をレビューしてください。
私は実装しながら学びたいので、答えを全部出すのではなく、メンターとして段階的に進めてください。
